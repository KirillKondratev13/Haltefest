# Отчет о ходе разработки (2026-02-23)

## Что было сделано
1.  **Реструктуризация в Monorepo**: Проект переведен на структуру монорепозитория. 
    - Текущий бэкенд перенесен в `services/backend`.
    - Созданы сервисы `services/parser` (Go) и `services/ml` (Python).
2.  **Инфраструктура**: Обновлен `docker-compose.yml`. Добавлены:
    - **Kafka**: Для обмена событиями между сервисами.
    - **DragonflyDB**: Как высокопроизводительная замена Redis для хранения извлеченного текста.
3.  **Parser Service (Go)**:
    - Реализован параллельный воркер (worker pool).
    - Поддерживаемые форматы: PDF (через `ledongthuc/pdf`), DOCX (ручной парсинг OOXML для надежности) и TXT.
    - Интегрирована детекция языка через `whatlanggo` (фильтрация только английских текстов).
    - Реализовано сохранение текста в Redis и уведомление ML-сервиса через Kafka.
4.  **ML Service (Python)**:
    - Реализован воркер на Python для классификации текста.
    - Добавлен мок для инференса модели NeoBERT (выбор случайного тега).
    - Реализовано обновление статуса и тега в базе данных PostgreSQL.
5.  **Интеграция Бэкенда**:
    - Бэкенд теперь отправляет событие `files-to-parse` в Kafka сразу после успешной загрузки файла в SeaweedFS.
6.  **Конфигурация**: Обновлен `.gitignore` для корректной работы с несколькими языками и вложенными сервисами.

## Трудности, с которыми столкнулись
- **API библиотек Go**: Библиотеки `pdfcpu` и `godocx` оказались больше ориентированы на создание/редактирование файлов, чем на потоковое извлечение текста. Был осуществлен переход на более простые и надежные решения (прямой парсинг XML для DOCX и `ledongthuc/pdf` для PDF).
- **Версионность Go**: Для работы некоторых новых библиотек потребовалось обновить версию Go до 1.24+ в Docker-файлах.
- **Интеграция Kafka**: Настройка корректного взаимодействия бэкенда с Kafka потребовала добавления интерфейсов в слое хендлеров, чтобы избежать цикличных зависимостей.

## С чего начать в следующий раз
1.  **Сквозное тестирование (E2E)**:
    - Запустить весь стек через `docker-compose up --build`.
    - Загрузить файл через UI/API и проследить его путь: `Backend -> Kafka -> Parser -> Redis -> Kafka -> ML -> Postgres`.
2.  **Замена мока в ML**: Подключить реальную модель NeoBERT вместо случайного выбора тегов.
3.  **Обработка ошибок**: Протестировать сценарии с битыми файлами и падениями Kafka/Redis.
4.  **Фронтенд**: Обновить UI для отображения тегов и статуса обработки в реальном времени (например, через HTMX polling).
